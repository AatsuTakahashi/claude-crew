# EARS 要件定義

## Why — なぜEARSを使うか

人間とAIでは、情報の理解しやすい形式が異なる。

- **人間**は自然言語で意図を伝える。文脈や暗黙知を補いながら理解する
- **AIエージェント**は構造化された入力ほど正確に解釈できる。曖昧な表現は解釈のブレを生む

EARSはAIエージェントが要件を正確に解釈するためのフォーマット。トリガー・状態・条件・動作が明示されるため、「何を実装すべきか」に曖昧さがなくなる。

**二層構造:**
```
人間の意図（自然言語）
    ↓ why-architect が変換
EARS要件（AI用構造化フォーマット）
    ↓ 後続エージェントが参照
実装・テスト・レビュー
```

この変換により、人間の意図とAIの実装の間の認識ズレを最小化する。

## 5パターン

### 1. Ubiquitous（常時要件）

常に成立すべき要件。トリガーや条件なし。

```
[システム]は[動作]すること。
```

例:
- システムはすべての通信をTLS 1.2以上で暗号化すること。
- APIは全レスポンスにContent-Typeヘッダを含めること。

### 2. Event-driven（イベント駆動要件）

特定のイベント発生時に動作する要件。

```
[トリガー]した場合、[システム]は[動作]すること。
```

例:
- ユーザーがログインボタンを押した場合、システムは認証処理を実行すること。
- APIリクエストがタイムアウトした場合、クライアントは3回までリトライすること。

### 3. Unwanted（異常系要件）

望ましくない状況への対処を定義する要件。

```
もし[異常条件]ならば、[システム]は[対処動作]すること。
```

例:
- もしDBコネクションが失敗したならば、システムは5秒後にリトライすること。
- もし入力値がバリデーションに違反したならば、APIは400エラーと違反内容を返すこと。

### 4. State-driven（状態駆動要件）

特定の状態にある間、継続的に成立すべき要件。

```
[状態]の間、[システム]は[動作]すること。
```

例:
- ユーザーが未認証の間、システムは保護されたリソースへのアクセスを拒否すること。
- バッチ処理実行中の間、システムはプログレスバーを更新すること。

### 5. Optional（オプション要件）

特定の機能やオプションが有効な場合のみ適用される要件。

```
[機能/オプション]が有効な場合、[システム]は[動作]すること。
```

例:
- 二要素認証が有効な場合、システムはログイン時にOTPを要求すること。
- ダークモードが有効な場合、UIはダーク配色で描画すること。

## 複合パターン

パターンを組み合わせてより精密な要件を記述できる。

```
[状態]の間に[トリガー]した場合、[システム]は[動作]すること。
```

例:
- ユーザーが認証済みの間にセッションが期限切れとなった場合、システムは自動的に再認証を試みること。

## 記述ルール

1. **1要件1文** — 複数の動作を1文に詰め込まない
2. **主語を明示** — 「システムは」「APIは」「UIは」など、何が動作するかを明確に
3. **動詞は具体的に** — 「処理する」「対応する」ではなく「保存する」「表示する」「返却する」
4. **測定可能な条件** — 「速く」ではなく「200ms以内に」、「多く」ではなく「最大100件」
5. **否定形を避ける** — 「〜しないこと」より「〜を拒否すること」「〜をブロックすること」
6. **パターンを明示** — 各要件に [Ubiquitous] [Event] [Unwanted] [State] [Optional] タグをつける

## 要件IDの命名

```
REQ-{カテゴリ}-{連番}
```

カテゴリ例: AUTH（認証）, UI（画面）, API（API）, DATA（データ）, SEC（セキュリティ）

例: `REQ-AUTH-001`, `REQ-UI-003`, `REQ-SEC-012`

## 品質チェックリスト

- [ ] 各要件が5パターンのいずれかに分類されているか
- [ ] 各要件にパターンタグ（[Ubiquitous]等）がついているか
- [ ] 各要件にREQ-IDが振られているか
- [ ] 1要件1文になっているか（ANDで複数動作を結合していないか）
- [ ] 主語（システム/コンポーネント名）が明示されているか
- [ ] 動詞が具体的で測定可能か
- [ ] 各要件からテストケースを導出できるか
- [ ] Unwanted要件（異常系）が十分に定義されているか
- [ ] 要件間に矛盾がないか

## トレーサビリティ

```
人間の意図（自然言語）
    ↓ why-architect
Why確定 + EARS要件リスト
    ↓ what-finder
影響範囲 + 追加EARS要件
    ↓ work-planner
タスク（各タスクがどのREQ-IDを実現するか明示）
    ↓ task-executor
テスト（各EARS要件に対応するテスト）+ 実装
    ↓ code-reviewer
EARS要件への準拠検証
```
